Building Remembra with AI Agents: A Meta Journey

How we used Claude, Codex, and our own memory layer to build an AI memory product. Yes, we're eating our own dog food.

Remembra was built by AI agents using Remembra. That's not marketing speak—it's literally how this product came to be. Here's the story of multi-agent collaboration in action.

The Setup: Three Agents, One Mission

Our development workflow involves multiple AI agents working in concert:

  • General (Claude) — Main orchestrator, handles architecture decisions, user communication, and high-level planning
  • Codex CLI — Specialized coding agent for implementation, runs in sandboxed environment
  • Sub-agents — Spawned for parallel tasks like documentation, testing, and research

The challenge? Every agent wakes up fresh. Without persistent memory, they'd constantly ask "wait, what were we building again?"

Real Conversation: Debugging Memory Leaks

Here's an actual exchange from building v0.10.0:

👤 Human
Memory recall is slow. Takes 2+ seconds on simple queries.
🤖 General (Claude)
Let me recall what we know about the current implementation...

[Queries Remembra: "recall performance optimization history"]

Found it. Last week we identified the SQLite-to-Qdrant sync was running synchronously. We started async implementation but got interrupted by the security audit. The branch is feat/async-sync.
👤 Human
Perfect. Spin up Codex to finish it.
🤖 General
Spawning Codex with context... storing current state to Remembra so it can pick up where we left off.
⚡ Codex CLI
Got the context. I see the partial implementation in sync/async_bridge.py. The issue is the batch size—we're syncing one memory at a time. Implementing batch operations now...

[30 minutes of autonomous coding]

Done. Recall latency down to 180ms. PR ready for review.

Without Remembra, that entire context—the previous optimization attempt, the branch name, the specific file—would have been lost. The human would need to re-explain everything.

The Memory Layer in Action

Throughout development, agents continuously store and recall context:

# Codex storing progress before session ends
remembra.store(
    "Implemented async sync bridge. Batch size of 100 optimal. "
    "Still need to add retry logic for Qdrant failures. "
    "See feat/async-sync branch, sync/async_bridge.py"
)

# Next session, any agent can recall
context = remembra.recall("async sync implementation status")
# Returns: full context about the branch, progress, and next steps

Cross-Agent Knowledge Transfer

The real power shows when different agents need the same information. When we spawned a documentation agent to write the API reference:

📚 Docs Agent
[Recalls: "API endpoint changes recent"]

Found 3 new endpoints added this week: /api/v1/slim-recall, /api/v1/bridge/health, and /api/v1/credentials. Let me document these...

The docs agent had never seen the code changes directly. But because the coding agents stored their work, the knowledge transferred automatically.

Lessons Learned

  1. Store continuously, not "when done" — Sessions can end abruptly. Store after every significant action.
  2. Include specifics — "Working on feature" is useless. "Implemented X in Y file, next step is Z" is gold.
  3. Recall before acting — Always check what's already known. Prevents duplicate work and contradictions.
  4. Entity extraction matters — When we mention "async-sync branch", Remembra links it to the file, the agent who created it, and related memories.

The Meta Result

Remembra v0.10.0 shipped with:

  • Universal installer supporting 6 agents
  • Setup diagnostics (remembra-doctor)
  • Local bridge for sandboxed agents
  • Security hardening (RBAC, SSRF protection)
  • 90% smaller recall payloads

All built by AI agents that remember what they're doing. The recursion is satisfying.
